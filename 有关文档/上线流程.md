# 上线流程

## 后端部署

后端使用 Spring Boot 打包部署。`mvn package` 只负责生成可运行产物，不会自动切换到 `prod` 配置。  
真正决定加载哪个配置的是启动时的 profile 来源，包括：

- `--spring.profiles.active=prod`
- `-Dspring.profiles.active=prod`
- `SPRING_PROFILES_ACTIVE=prod`
- 外部配置文件或容器编排中注入的 profile

### 打包

在 `backend` 目录下执行：

```bash
mvn clean package -DskipTests
```

打包后的产物通常位于：

```bash
backend/target/backend-1.0.0.jar
```

### 配置文件准备

仓库内只保留模板文件，生产环境需要单独准备运行时配置：

- `backend/src/main/resources/application.yml`：公共配置
- `backend/src/main/resources/application-dev.example.yml`：开发环境模板
- `backend/src/main/resources/application-prod.example.yml`：生产环境模板

上线前把 `application-prod.example.yml` 复制为实际运行文件 `application-prod.yml`，并替换数据库、Redis、对象存储等真实配置。

### 启动示例

显式指定生产 profile：

```bash
java -jar backend/target/backend-1.0.0.jar --spring.profiles.active=prod
```

也可以用 JVM 参数：

```bash
java -Dspring.profiles.active=prod -jar backend/target/backend-1.0.0.jar
```

如果通过环境变量启动：

```bash
export SPRING_PROFILES_ACTIVE=prod
java -jar backend/target/backend-1.0.0.jar
```

### 为什么会自动进 prod

如果部署后看到后端使用的是 `prod` 配置，通常不是 `mvn package` 导致的，而是启动环境里已经把 profile 设成了 `prod`。优先检查：

1. 启动脚本里是否带了 `--spring.profiles.active=prod`
2. 环境变量里是否设置了 `SPRING_PROFILES_ACTIVE=prod`
3. `systemd`、Docker、K8s、CI/CD 脚本里是否注入了 `prod`
4. 服务器上的外部 `application-prod.yml` 是否生效

### 排查顺序

当配置看起来“自动变成 prod”时，按下面顺序查：

1. 先看实际启动命令
2. 再看环境变量
3. 再看 `systemd`、Docker、K8s、CI/CD 配置
4. 最后看外部 `application-prod.yml`
