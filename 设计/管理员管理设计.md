# 管理员管理设计

## 1. 设计前提

当前后台管理员体系基于以下 4 张表：

- `admin_user`
  - 管理员账号主体，承载登录账号、显示名称、密码、状态、默认项目、最后登录时间。
- `admin_role`
  - 角色定义，只包含角色编码、角色名称、状态。
- `admin_user_role`
  - 管理员与角色的多对多关系。
- `admin_project_access`
  - 管理员可访问的项目范围，控制能进入哪些业务域。

这意味着当前最适合采用的不是“菜单/按钮/接口全表驱动权限”，而是：

1. 角色权限
   - 由 `role_code` 决定系统级能力。
2. 项目范围
   - 由 `admin_project_access` 决定能操作哪些业务项目。

所以第一阶段推荐做“角色 + 项目范围”两层权限模型，不额外新增权限表。

## 2. 权限模型

### 2.1 两层权限

第一层：系统角色权限

- 决定一个管理员“能做什么”
- 例如是否能管理管理员、是否能管理用户、是否能管理项目业务内容

第二层：项目范围权限

- 决定一个管理员“能在哪个项目里做”
- 例如同样都是内容运营，只能进 `kyzz`，不能进 `kysx`

最终权限 = 角色能力集合 ∩ 项目访问范围

### 2.2 推荐角色划分

基于现有表结构，建议先内置以下角色编码：

- `SUPER_ADMIN`
  - 全局最高权限
  - 可管理所有管理员、角色、项目授权、业务数据
- `SYSTEM_ADMIN`
  - 系统管理员
  - 可管理普通管理员、角色分配、项目授权
  - 不允许修改超级管理员核心信息，不允许给自己提权为超级管理员
- `PROJECT_ADMIN`
  - 项目管理员
  - 可管理自己有权限项目下的业务内容、项目用户、项目运营配置
  - 不能管理后台管理员和角色
- `CONTENT_OPERATOR`
  - 内容运营
  - 可维护题库、分类、标签、内容上下架等
  - 只能操作被授权项目
- `DATA_VIEWER`
  - 只读查看
  - 可查看工作台、报表、列表详情
  - 不可新增、修改、删除

### 2.3 角色与菜单建议映射

- `SUPER_ADMIN`
  - 工作台
  - 用户管理
  - 管理员管理
  - 角色管理
  - 全部项目业务菜单
- `SYSTEM_ADMIN`
  - 工作台
  - 用户管理
  - 管理员管理
  - 角色管理
  - 被授权项目的业务菜单
- `PROJECT_ADMIN`
  - 工作台
  - 被授权项目的业务菜单
- `CONTENT_OPERATOR`
  - 工作台
  - 被授权项目下的内容维护菜单
- `DATA_VIEWER`
  - 工作台
  - 被授权项目下的只读菜单

## 3. 管理员管理应该具备的功能

### 3.1 管理员账号功能

- 新增管理员
  - 录入账号、显示名称、初始密码、默认项目、状态
- 编辑管理员
  - 修改显示名称、默认项目、状态
- 重置密码
  - 管理员列表中发起重置，不直接展示原密码
- 启用/禁用管理员
  - 禁用后不可登录
- 查看最后登录时间
- 查看角色列表
- 查看项目授权范围

### 3.2 角色分配功能

- 为管理员分配多个角色
- 移除角色
- 查看角色状态
- 限制危险操作
  - 非超级管理员不可分配 `SUPER_ADMIN`
  - 管理员不可禁用自己
  - 管理员不可移除自己唯一的高权限角色

### 3.3 项目授权功能

- 为管理员勾选可访问项目
- 设置默认项目
- 禁用单个项目访问权限
- 限制默认项目必须属于已授权项目
- 支持快速复制已有管理员的项目授权

### 3.4 安全相关功能

基于现有表结构，第一阶段建议做：

- 修改本人密码
- 管理员重置他人密码
- 查看最后登录时间

当前不建议直接做独立页面的功能：

- 登录日志列表
- 操作审计日志
- 细粒度按钮权限配置

原因是现有表结构没有对应日志或权限映射表，做出来会先依赖硬编码，后期返工会比较大。

## 4. 建议的后台页面

### 4.1 通用管理

#### 1. 个人资料页

路径建议：`/system/profile`

功能：

- 查看本人账号信息
- 修改显示名称
- 修改默认项目
- 修改本人密码
- 查看最近登录时间

说明：

- 这页已经完成首版

#### 2. 管理员管理页

路径建议：`/system/admins`

这是核心页面，建议做成“列表页 + 抽屉详情”。

列表字段建议：

- 管理员 ID
- 登录账号
- 显示名称
- 角色标签
- 默认项目
- 可访问项目数
- 状态
- 最后登录时间
- 创建时间

筛选项建议：

- 账号/显示名称关键词
- 状态
- 角色
- 项目

行操作建议：

- 查看详情
- 编辑资料
- 分配角色
- 配置项目权限
- 重置密码
- 启用/禁用

#### 3. 角色管理页

路径建议：`/system/admin-roles`

虽然现在没有“角色-权限表”，但角色页仍然值得做，用于统一维护角色定义和说明。

列表字段建议：

- 角色编码
- 角色名称
- 状态
- 关联管理员数
- 创建时间

功能建议：

- 新增角色
- 编辑角色名称
- 启用/禁用角色
- 查看该角色对应的系统能力说明

实现建议：

- 第一阶段角色能力先在后端用枚举或配置映射
- 页面上展示“此角色拥有的能力说明”
- 暂不提供按钮级勾选权限编辑器

#### 4. 管理员授权详情抽屉

可挂在 `/system/admins` 内，不一定单独起路由。

建议分 4 个 tab：

- 基本信息
- 角色分配
- 项目权限
- 安全操作

每个 tab 职责：

- 基本信息
  - 账号、显示名称、状态、默认项目、最近登录
- 角色分配
  - 当前角色、添加角色、移除角色
- 项目权限
  - 授权项目勾选、单项目启停、默认项目设置
- 安全操作
  - 重置密码、风险提示

### 4.2 业务管理

业务管理菜单继续基于 `projectCode` 渲染，不单独为管理员管理新增根级模块。

建议：

- `PROJECT_ADMIN`、`CONTENT_OPERATOR`、`DATA_VIEWER` 只显示自己有权限项目的菜单
- 同一个业务页内部仍应继续校验项目访问权限，不能只依赖前端隐藏菜单

## 5. 页面实施顺序

建议按以下顺序落地：

1. `个人资料`
   - 已完成
2. `管理员管理列表页`
   - 先把账号列表、启停、重置密码、编辑资料做起来
3. `管理员授权抽屉`
   - 接角色分配和项目授权
4. `角色管理页`
   - 统一沉淀角色说明和角色状态

这样做的原因：

- 先完成最常用的管理员维护流程
- 不会因为没有细粒度权限表而卡住
- 后续即使新增 `role_permission`、`permission_point` 等表，也能平滑扩展

## 6. 后端接口建议

第一阶段建议补以下接口：

- `GET /api/admin/users`
  - 管理员列表
- `POST /api/admin/users`
  - 新增管理员
- `PUT /api/admin/users/{id}`
  - 编辑管理员基础资料
- `PUT /api/admin/users/{id}/status`
  - 启用/禁用管理员
- `PUT /api/admin/users/{id}/password/reset`
  - 重置管理员密码
- `GET /api/admin/roles`
  - 角色列表
- `POST /api/admin/users/{id}/roles`
  - 批量设置管理员角色
- `GET /api/admin/users/{id}/projects`
  - 查看管理员项目权限
- `POST /api/admin/users/{id}/projects`
  - 批量设置管理员项目权限

说明：

- 管理员自身资料和改密继续放在 `/api/admin/auth/*`
- 管理别人账号的能力应单独进入 `/api/admin/users/*`

## 7. 风险控制规则

页面和接口层都要加以下保护：

- 不能禁用自己
- 不能删除或禁用最后一个超级管理员
- 非超级管理员不能分配超级管理员角色
- 默认项目必须属于该管理员已授权项目
- 被禁用角色不应继续赋给新管理员
- 被禁用管理员登录时直接拒绝

## 8. 当前结论

基于现有表结构，管理员管理第一阶段不应该追求“超细权限平台”，而应该先做：

- 个人资料
- 管理员管理
- 角色管理
- 项目授权

权限模型用：

- `角色控制能做什么`
- `项目权限控制在哪做`

这套方案和你当前表结构完全匹配，开发成本低，也不会把后台复杂度一下拉高。
